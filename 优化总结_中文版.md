## 🛠️ 技术实现

### 修改的文件

**1. cooldown.py** - 新增状态管理系统
```python
# 新增：记录是否已推送过通知
can_trade_time: Optional[datetime]  # 何时可以开单
should_notify: bool = False         # 是否应该推送

# 新增：状态文件管理函数
load_cooldown_notify_state()    # 读取通知状态
save_cooldown_notify_state()    # 保存通知状态
reset_cooldown_notify_state()   # 重置通知状态（冷静期结束时）
```

**2. strategy.py** - 改进推送决策
```python
# 原逻辑：每次都推送
if cooldown.triggered:
    return TradeResult(action="cooldown", ...)

# 新逻辑：仅首次推送
if cooldown.triggered:
    if cooldown.should_notify:  # 首次?
        action = "cooldown"     # 推送
    else:
        action = "none"         # 不推送
    return TradeResult(action=action, ...)
```

### 自动状态管理

**状态文件：** `cooldown_notify_state.json`

```json
{
  "notified": false,           // false=未通知, true=已通知
  "triggered_at": null,        // 首次触发的时间
  "notify_count": 0            // 推送次数
}
```

**自动流程：**
```
首次进入冷静期
  ↓ 保存 notified=true
  ↓ 返回 should_notify=True
  ↓ 推送1条通知 ✅
  
30分钟后再检查
  ↓ 读取 notified=true
  ↓ 返回 should_notify=False
  ↓ 不推送 ✅

48小时后冷静期结束
  ↓ 自动重置 notified=false
  ↓ 准备下一轮
```

## 💡 常见问题

### Q1: 为什么要用状态文件？

**A:** 因为脚本是无状态的（每30分钟运行一次后退出），需要持久化存储来记录是否已推送过。用JSON文件最简单可靠。

### Q2: 状态文件会不会丢失？

**A:** 不会。文件持久化存储在硬盘上。即使重启也不会丢失。脚本会自动加载和更新。

### Q3: 如何手动重置冷静期状态？

**A:** 
```bash
# 方式1：删除文件（脚本会自动重建）
rm cooldown_notify_state.json

# 方式2：Python命令
python3 -c "from cooldown import reset_cooldown_notify_state; reset_cooldown_notify_state()"
```

### Q4: 是否会影响交易逻辑？

**A:** 完全不影响。这只改进了推送策略，交易执行逻辑和冷静期检查完全保持不变。

### Q5: 如何回滚到旧版本？

**A:** 直接恢复原始代码即可。所有改动向后兼容，不会有任何问题。
